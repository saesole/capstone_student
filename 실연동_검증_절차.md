# 실연동 검증 절차 (sim_mode:=false) — ND1 캡스톤

> ⚠️ 이 검증은 ROS2·Gazebo·Nav2가 있는 **컨테이너 안에서만** 수행 가능.
> 단계별로 "기대 결과"가 나오는지 확인하고, 안 나오면 아래 트리아지 표로 진단.

## 0. 사전 조건
- `docker compose up --build` 로 컨테이너 기동, noVNC(localhost:8080) 접속
- `.env` 에 `GROQ_API_KEY` 입력(없어도 폴백으로 진행 가능)
- `cd ~/ros2_ws && colcon build && source install/setup.bash`

## 1. 단계 A — sim_mode 먼저 통과 (B/C/Gazebo 불필요)
```bash
ros2 launch nd1_capstone bringup.launch.py sim_mode:=true     # 터미널 1
python3 ~/ros2_ws/src/.../scripts/smoke_test.py               # 터미널 2
```
기대: 콘솔에 `SMOKE TEST: PASS`, 상태가 NAVIGATING→GRASPING→TRANSPORTING→PLACING→DONE.
→ 통과해야 ②로. 실패 시 코드/토픽 문제(아래 T1~T3).

## 2. 단계 B — Gazebo + SLAM + Nav2 기동 (map 프레임 + navigate_to_pose 확보)

> ⚠️ **핵심**: Nav2 단독(`nav2_bringup navigation_launch.py`)은 planner/controller/costmap만
> 올리고 **map→odom 공급원(SLAM 또는 AMCL)을 포함하지 않는다.** 그러면 TF에 `map`
> 프레임이 없어 global_costmap이 다음 에러를 무한 반복하고 NAVIGATING에서 막힌다:
> `Invalid frame ID "map" passed to canTransform ... frame does not exist` (→ 트리아지 T3').
> 따라서 **SLAM(권장) 또는 localization 을 반드시 함께 띄운다.**

> ⚠️⚠️ **성능(RTF) — 가장 먼저 확인.** 소프트웨어 렌더(llvmpipe)면 RTF가 ~0.037까지 떨어져
> controller 사망·TF 끊김·map 미수신·미션 타임아웃이 **연쇄로** 발생한다(실측됨).
> Windows+WSL에서 검증된 정답 시뮬 기동(RTF 0.99 달성, GPU=D3D12 Intel Xe):
> ```bash
> ros2 launch turtlebot4_ignition_bringup turtlebot4_ignition.launch.py \
>   model:=lite world:=warehouse rviz:=false \
>   gz_args:="-s -r --render-engine ogre --headless-rendering"
> ```
> 4요소: `--render-engine ogre`(ogre2 copyTo 크래시 회피)·`-s`(서버만, sun 충돌 회피)·
> `--headless-rendering`·`model:=lite`(카메라 제거). 사전: WSLg override로 기동 + `glxinfo`에 D3D12 확인.
> 측정: `ign topic -e -t /stats | grep -m1 real_time_factor` → **0.5 미만이면 아래 절차 진행 무의미**, 먼저 GPU부터.
> 상세: `WSLg_렌더_가이드.md`.

```bash
# 터미널 1: 시뮬 (※ GPU 가속 시 위 정답 명령 사용 권장)
ros2 launch turtlebot4_ignition_bringup turtlebot4_ignition.launch.py

# 터미널 2: SLAM — map→odom 발행 (사전 맵 불필요, 시뮬 시연 최단경로)
ros2 launch turtlebot4_navigation slam.launch.py

# 터미널 3: Nav2 — navigate_to_pose 액션 서버
ros2 launch turtlebot4_navigation nav2.launch.py
```

**map 프레임이 실제로 생겼는지 확인 (이게 핵심 검증):**
```bash
ros2 run tf2_ros tf2_echo map base_link     # 좌표가 흘러나오면 정상(에러 사라짐)
ros2 topic echo /map --once                 # OccupancyGrid 수신 확인
ros2 node list | grep -E "slam|amcl"        # map→odom 공급원이 떠 있는지
ros2 action list | grep navigate_to_pose    # 액션 서버 존재 확인
```
기대: `tf2_echo map base_link` 가 변환을 출력하고, `/navigate_to_pose` 액션이 보임.

**대안 — 사전 맵을 쓰는 경우(SLAM 대신 AMCL):**
```bash
ros2 launch turtlebot4_navigation localization.launch.py map:=/경로/warehouse.yaml
ros2 launch turtlebot4_navigation nav2.launch.py
# → RViz "2D Pose Estimate"로 초기 위치를 찍어야 AMCL이 map→odom 발행 시작
```

## 3. 단계 C — 실연동 기동

방법 ①(권장, 터미널 최소화) — 패치된 `bringup.launch.py`가 SLAM+Nav2+4노드를 한 번에:
```bash
# 터미널 1: 시뮬 (그대로)
ros2 launch turtlebot4_ignition_bringup turtlebot4_ignition.launch.py

# 터미널 2: SLAM + Nav2 + 캡스톤 4노드 통합 기동
ros2 launch nd1_capstone bringup.launch.py \
  sim_mode:=false slam:=true nav2:=true \
  -p node_c_grasp.world_name:=<월드명> -p node_c_grasp.box_model:=<박스모델명>

# 터미널 3: 스모크
python3 .../scripts/smoke_test.py "A구역 박스를 B구역으로 옮겨줘" 60
```

방법 ②(사전 맵/AMCL) — SLAM 대신 localization + 맵 경로:
```bash
ros2 launch nd1_capstone bringup.launch.py \
  sim_mode:=false localization:=true map:=/경로/warehouse.yaml nav2:=true
# → RViz "2D Pose Estimate"로 초기 위치 지정 필수
```

> 통합 기동 시 SLAM/Nav2가 완전히 올라오는 데 수 초가 걸린다. `node_b_nav`는
> navigate_to_pose 서버를 기다리므로 순서 문제는 없지만, 첫 NAVIGATING 직전
> 단계 B의 `tf2_echo map base_link` 로 map 프레임을 먼저 확인하면 안전하다.

기대 로그 순서:
```
[A] LLM 파싱 성공 (또는 폴백)
[FSM:UNDOCKING] undock 요청        ← 도킹 상태면 Node B가 자동 undock(미도킹이면 생략)
[B] dock 결과: 성공
[FSM:NAVIGATING] 이동 요청 → (1.50, 0.50)
[B] 이동 결과: 성공
[FSM:GRASPING] grasp 요청 → (1.50, 0.50)
[C] grasp IK 해 q=[...] (target=(0.35,0.25))
[C] 결과: 성공
[FSM:TRANSPORTING] 이동 요청 → (2.50, -1.00)
[B] 이동 결과: 성공
[FSM:PLACING] place 요청 → (2.50, -1.00)
[C] 결과: 성공
[FSM:DOCKING] 미션 완료 → 재도킹    ← auto_redock=true(기본). false면 바로 DONE
[B] dock 결과: 성공
[FSM:DOCKING] 재도킹 완료 → DONE
SMOKE TEST: PASS
```
> ※ 로봇은 도킹 중엔 Nav2 주행이 막힌다. v3부터 Coordinator가 이동 전 자동 undock,
>   미션 후 자동 재도킹한다(`auto_redock:=false`로 끄기 가능). 하 난이도(linear_orchestrator)는
>   undock 미포함 → 실연동 시 수동 undock 필요:
>   `ros2 action send_goal /undock irobot_create_msgs/action/Undock {}`

## 4. 실패 트리아지

| 코드 | 증상 | 원인 | 조치 |
|------|------|------|------|
| T1 | smoke_test가 명령 후 무반응 | 토픽 이름/타입 불일치 | `ros2 topic list`, `ros2 topic echo /mission` 로 흐름 확인 |
| T2 | sim_mode인데 DONE 안 됨 | colcon build 누락/소스 미반영 | 재빌드 후 `source install/setup.bash` |
| T3 | NAVIGATING에서 정지 | Nav2 서버 없음 | `ros2 action list`에 navigate_to_pose 확인, 단계 B 재기동 |
| T3' | `Invalid frame ID "map" ... frame does not exist` (costmap 타임아웃 반복) | map→odom 공급원 없음(SLAM/AMCL 미기동) — Nav2 단독만 띄움 | `slam:=true`(또는 `localization:=true`+`map`) 추가. `ros2 run tf2_ros tf2_echo map base_link`로 변환 확인, `ros2 node list \| grep -E "slam\|amcl"` |
| T4 | "Nav2 goal 거부" | 맵/AMCL 미초기화, 목표 비유효 | RViz "2D Pose Estimate"로 초기 위치 설정, 좌표가 맵 범위 내인지 확인 |
| T5 | GRASPING에서 IK 실패 | y-offset<0.20 또는 reach 초과 | grasp_y≥0.20 확인, arm_links 합 > target 거리 확인 |
| T6 | "ign service 응답 false" | world_name/box_model 불일치 | `ign model --list`, `ign service --list`로 실제 이름 확인 후 파라미터 수정 |
| T7 | Gazebo 화면 깨짐/크래시 | WSL OpenGL | `LIBGL_ALWAYS_SOFTWARE=1` 적용 확인(기본 내장됨) |
| T8 | LLM만 실패, 폴백 동작 | groq 키/모델/네트워크 | `.env` 키 확인, `GROQ_MODEL` 유효성(console.groq.com/docs/models) |
| T9 | UNDOCKING/DOCKING에서 멈춤 또는 로봇이 도크에서 못 나옴 | dock 액션/토픽 이름 불일치 또는 irobot_create_msgs 미설치 | `ros2 action list \| grep -i dock`, `ros2 interface show irobot_create_msgs/msg/DockStatus` 로 확인 후 Node B 파라미터(`undock_action`/`dock_action`/`dock_status_topic`) 교정. 임시로 `auto_redock:=false` + 수동 undock |
| T10 | controller 사망·TF 끊김·map 미수신·미션 타임아웃이 한꺼번에 | **RTF 폭락(소프트웨어 렌더)** — 단일 근본원인의 연쇄 | `ign topic -e -t /stats`로 RTF 확인. <0.1이면 WSLg+GPU+`--render-engine ogre`+`-s`+`model:=lite`(검증 조합, RTF 0.99). `WSLg_렌더_가이드.md` 참조 |
| T11 | `Ogre::UnimplementedException`(copyTo) / `sun already exists` | d3d12 + ogre2 미구현 / ogre1 GUI 조명 중복 | `--render-engine ogre` + `-s` 헤드리스 + `model:=lite` 로 회피 |
